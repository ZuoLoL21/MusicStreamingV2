# API Service User Database

> **⚠️ INTERNAL SERVICE ONLY**
> This service is NOT directly accessible to clients. All client requests must go through gateway_api (port 8080).
> This documentation is for backend developers and service-to-service communication reference only.

This document describes the RESTful API for the User Database microservice, which manages users, artists, albums, music, playlists, and tags.

## Access

**Accessed via gateway_api proxy routes** - Clients use `http://localhost:8080` with routes like `/users/*`, `/artists/*`, `/albums/*`, etc.

## Authentication Overview

The API uses Service JWT tokens for authentication (automatically added by gateway_api). Login and register endpoints are proxied through the gateway. Only `/health` and `/files/*` endpoints do not require authentication.

### Authentication Header

```http
Authorization: Bearer <service_jwt_token>
```

**Note:** Service JWTs are automatically generated by gateway_api when proxying client requests. Clients should never generate Service JWTs directly.

### Role-Based Access Control

The API implements three roles:
- **User**: Basic authenticated user
- **Member**: Artist member (can create music, albums)
- **Manager**: Artist manager (can manage artist, albums, music)
- **Owner**: Artist owner (can manage members, delete resources)

### HTTP Methods

- **PUT**: Create new resources
- **POST**: Update existing resources or perform actions
- **GET**: Retrieve resources
- **DELETE**: Remove resources

## Authentication Endpoints

### Register User

Register a new user account.

**Endpoint:** `PUT /login`

**Content-Type:** `multipart/form-data`

**Request Fields:**
| Field | Type | Required | Description |
|-------|------|----------|-------------|
| username | string | Yes | Unique username |
| email | string | Yes | User email address |
| password | string | Yes | User password |
| country | string | Yes | Country code (e.g., "US", "CA") |
| bio | string | No | User biography |
| image | file | No | Profile image file |

**Status Codes:**
- `201 Created`: User successfully registered
- `400 Bad Request`: Invalid request body
- `409 Conflict`: Username or email already exists

**Example Request (cURL):**
```bash
curl -X PUT http://localhost:8080/login \
  -F "username=johndoe" \
  -F "email=john@example.com" \
  -F "password=secret123" \
  -F "country=US" \
  -F "bio=Music lover"
```

---

### Login

Authenticate a user and obtain a JWT token.

**Endpoint:** `POST /login`

**Content-Type:** `application/json`

**Request Body:**
```json
{
  "email": "john@example.com",
  "password": "secret123"
}
```

**Status Codes:**
- `200 OK`: Login successful, returns JWT token
- `400 Bad Request`: Invalid request body
- `401 Unauthorized`: Invalid credentials

**Response:**
```json
{
  "access_token": "eyJhbGciOiJIUzI1NiIs...",
  "refresh_token": "eyJhbGciOiJIUzI1NiIs...",
  "user_uuid": "123e4567-e89b-12d3-a456-426614174000"
}
```

---

### Renew Token

Renew an expiring JWT token.

**Endpoint:** `POST /renew`

**Authentication:** Required (JWT)

**Status Codes:**
- `200 OK`: Token renewed successfully
- `401 Unauthorized`: Invalid or missing token

### Reorder Playlist Tracks

Reorder all tracks in a playlist by providing the desired order (Owner only).

**Endpoint:** `POST /playlists/{uuid}/reorder`

**Authentication:** Required (JWT + Owner)

**Content-Type:** `application/json`

**Request Body:**
```json
{
  "track_order": [
    "music-uuid-3",
    "music-uuid-1",
    "music-uuid-2"
  ]
}
```


**Path Parameters:**
- `uuid`: Playlist's unique identifier

**Query Parameters:**
| Field | Type | Required | Description |
|-------|------|----------|-------------|
| track_order | array of UUIDs | Yes | Complete ordered list of music UUIDs in desired sequence |

**Validation:**
- All UUIDs in `track_order` must be valid music UUIDs currently in the playlist
- Array length must match the current number of tracks in the playlist
- No duplicate UUIDs allowed
- No missing or extra tracks allowed

**Status Codes:**
- `200 OK`: Tracks reordered successfully
- `400 Bad Request`: Invalid UUIDs, count mismatch, or validation failed
- `401 Unauthorized`: Invalid or missing token
- `403 Forbidden`: User is not the owner
- `500 Internal Server Error`: Database error

**Notes:**
- This endpoint replaces the entire track order atomically
- Frontend should send the complete list in the desired final order
- Positions are automatically assigned as 0-indexed based on array order
